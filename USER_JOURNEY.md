# 사용자 여정 (User Journeys)

> `SPEC.md`에 정의된 기능들을, 실제 사용자가 화면에서 어떻게 쓰는지를 **시나리오 단위**로 정리한 문서입니다. 다음의 입력으로 쓰입니다:
>
> - 유닛 / 통합 / E2E (Playwright) 테스트의 시나리오 근거
> - **Playwright MCP를 통한 수동 회귀 테스트** 체크리스트
> - 새 기능 구현 후 "이 시나리오가 여전히 통과하는가?" 판단 기준
>
> 각 시나리오는 **Given / When / Then** 형식으로 작성하며, 구체 입력값·버튼 라벨을 포함해 테스터(또는 Playwright MCP)가 **그대로 따라 할 수 있게** 합니다.

## J1. 첫 방문 — 빈 WBS 맞이하기

**Given** 사용자가 이 앱을 처음 연다 (DB에 작업이 한 건도 없음).

**When** 브라우저로 `/` 에 접속한다.

**Then**
- "아직 작업이 없습니다. 첫 작업을 추가해 시작하세요" 안내 문구가 화면 중앙에 보인다.
- **+ 작업 추가** 버튼이 보인다.
- 툴바에 **CSV 내보내기 / CSV 불러오기** 버튼이 보인다 (비어 있어도 노출).

---

## J2. 첫 최상위 작업 추가

**Given** J1 상태.

**When**
1. **+ 작업 추가** 클릭
2. 모달에서 다음을 입력:
   - 제목: `기획 회의`
   - 담당자: `김PM`
   - 시작일: `2026-05-01`
   - 목표 기한: `2026-05-03`
3. **저장** 클릭

**Then**
- 모달이 닫힌다.
- 목록에 "기획 회의" 행이 한 건 추가된다.
  - 담당자: `김PM`
  - 상태 배지: `할 일`
  - 진행률: `0%`
  - 시작일·목표 기한: `2026-05-01 ~ 2026-05-03`
- "아직 작업이 없습니다..." 안내는 사라진다.

---

## J3. 하위 작업 추가 (계층 생성)

**Given** J2 완료 상태.

**When**
1. "기획 회의" 행의 **⋯ 메뉴** 열기 → **하위 작업 추가** 클릭
2. 제목 `아젠다 초안 작성` 입력 → **저장**

**Then**
- "기획 회의" 왼쪽에 **▼** 아이콘이 생긴다.
- 바로 아래에 한 단계 들여쓰기된 "아젠다 초안 작성" 행이 표시된다.
- 수정 모달을 다시 열어 보면 **상위 작업** 필드가 "기획 회의"로 채워져 있다.

---

## J4. 계층 접기/펼치기

**Given** J3 완료 상태.

**When** "기획 회의" 왼쪽의 **▼** 클릭.

**Then**
- 아이콘이 **▶**로 바뀐다.
- "아젠다 초안 작성" 행이 화면에서 숨는다.

**When** **▶** 를 다시 클릭.

**Then** 아이콘이 **▼**로 바뀌고 하위 행이 다시 보인다.

---

## J5. 진행률 100% → 상태 자동 "완료"

**Given** "아젠다 초안 작성" 존재. 진행률 0%, 상태 "할 일".

**When**
1. 행 클릭 → 수정 모달
2. 진행률 입력란에 `100` 입력
3. **저장**

**Then**
- 행의 진행률이 `100%`로 표시된다.
- 상태 배지가 **완료**로 자동 전환된다.
- 이 동작은 저장 시점이 아니라 **저장 후 목록 재렌더**에서 확인 가능하다.

---

## J6. 상태 배지 인라인 순환 전환

**Given** "기획 회의" 상태 "할 일".

**When** 목록에서 해당 행의 상태 배지 **"할 일"** 을 직접 클릭 → **"진행 중"** 선택.

**Then**
- 모달이 열리지 않는다.
- 배지가 즉시 **"진행 중"** 으로 바뀐다.
- 새로고침 후에도 상태는 유지된다.

**When** 같은 배지를 다시 클릭 → **"완료"** 선택.

**Then** 배지가 **"완료"** 로 바뀐다.

**When** 한 번 더 클릭 → **"할 일"** 선택.

**Then** 배지가 **"할 일"** 로 돌아간다. 세 상태 간 순환 전환이 모두 동작한다. 진행률은 이 전환으로 **자동으로 바뀌지 않는다** (완료로 바꿔도 100%가 되지 않고, 할 일로 되돌려도 0%가 되지 않음).

---

## J7. 작업 제목 수정

**Given** "기획 회의" 존재.

**When** 행 클릭 → 모달에서 제목을 `킥오프 미팅`으로 수정 → **저장**.

**Then** 목록 행의 제목이 즉시 "킥오프 미팅"으로 바뀐다.

---

## J8. 하위를 가진 작업 삭제

**Given** "기획 회의"(하위 1개: "아젠다 초안 작성") 존재.

**When**
1. "기획 회의" 행의 **⋯ 메뉴** → **삭제**
2. 다이얼로그 문구 확인: "이 작업과 하위 작업 1개가 모두 삭제됩니다. 계속할까요?"
3. **확인** 클릭

**Then**
- "기획 회의"와 "아젠다 초안 작성" 두 행이 모두 목록에서 사라진다.
- 새로고침 후에도 두 행은 복구되지 않는다.

---

## J9. 목표 기한 지남 (Overdue) — 목록 뷰

**Given**
- 오늘 날짜가 `2026-05-10` 이라고 가정 (또는 시스템 시각이 그 이후).
- "킥오프 미팅" 목표 기한: `2026-05-03`, 상태: "진행 중".

**When** 목록 뷰에서 해당 행을 본다.

**Then**
- 목표 기한 날짜 텍스트가 **경고색(빨강 계열)** 으로 표시된다.
- 날짜 옆에 **"지남"** 배지가 붙어 있다.

**When** 상태를 "완료"로 바꾼다.

**Then** 경고색과 배지가 즉시 사라진다.

---

## J10. CSV 내보내기

**Given** 목록에 작업이 3개 이상 있음 (최소 한 건은 하위 작업 포함).

**When** 툴바의 **CSV 내보내기** 클릭.

**Then**
- 파일 다운로드가 시작된다 (파일명 예: `wbs-2026-04-24.csv`).
- 파일을 열어 보면:
  - 첫 줄 헤더: `제목, 설명, 담당자, 상태, 진행률, 시작일, 목표 기한, 상위 작업 제목`
  - 데이터 행 수가 화면 작업 수와 일치.
  - 하위 작업 행의 "상위 작업 제목" 열에 부모의 제목이 채워져 있다.

---

## J11. CSV 가져오기 — 정상

**Given**
- 목록 뷰에 기존 작업 2개 존재.
- 유효한 CSV 파일(아래) 준비:
  ```csv
  제목,설명,담당자,상태,진행률,시작일,목표 기한,상위 작업 제목
  리서치,경쟁사 조사,이대리,할 일,0,2026-05-04,2026-05-07,
  리서치 요약,,이대리,할 일,0,2026-05-08,2026-05-08,리서치
  리뷰 미팅,,김PM,할 일,0,2026-05-10,2026-05-10,
  ```

**When**
1. 툴바 **CSV 불러오기** → 위 파일 선택
2. 미리보기 화면 확인: "3개 작업을 추가합니다. 제외 0건"
3. **적용** 클릭

**Then**
- 목록에 3개 작업이 추가된다 (총 5개).
- "리서치 요약"은 "리서치" 하위로 들어가 있다.
- 기존 2개는 그대로 유지된다.

---

## J12. CSV 가져오기 — 부분 오류

**Given**
- 아래 CSV 준비 (3행 중 1행에 제목 누락, 1행에 상위 매칭 실패):
  ```csv
  제목,설명,담당자,상태,진행률,시작일,목표 기한,상위 작업 제목
  문서화,,김PM,할 일,0,2026-05-11,2026-05-12,
  ,설명만 있음,,,,,,
  QA,,박테스터,할 일,0,2026-05-13,2026-05-14,존재하지않는부모
  ```

**When** **CSV 불러오기** → 파일 선택 → 미리보기 확인.

**Then (미리보기 화면)**
- "2개 작업을 추가합니다. 제외 1건" 표시.
- 제외 사유 목록에 "2행: 제목 누락" 이 나온다.
- "QA"는 "상위 매칭 실패 → 최상위로 처리" 안내가 붙는다 (제외 아님, 경고).

**When** **적용** 클릭.

**Then**
- "문서화", "QA" 두 행이 목록에 추가된다 ("QA"는 최상위).
- "설명만 있음" 행은 들어오지 않는다.

---

## J13. 간트 뷰로 전환

**Given** 목록 뷰에 최소 2개의 작업이 있고, 적어도 하나는 시작일/목표 기한이 모두 설정됨.

**When** 상단 토글 → **간트** 선택.

**Then**
- 화면이 좌우 2단 구조로 바뀐다.
- 좌측: 작업 트리(목록 뷰와 동일한 계층·들여쓰기).
- 우측: 가로 날짜 그리드. 눈금은 **주 단위**.
- **오늘 날짜에 세로 강조선**이 그려져 있다.
- 날짜가 설정된 작업은 해당 구간에 가로 막대가 있다.
- 날짜가 비어 있는 작업 행은 "— 일정 없음 —" 문구만 표시된다.

---

## J14. 간트 막대의 진행률 채움 확인

**Given** "킥오프 미팅" 진행률 60%, 기간 `2026-05-01 ~ 2026-05-10`.

**When** 간트 뷰에서 해당 막대를 본다.

**Then**
- 막대 **왼쪽 60%** 구간이 진한 색으로 채워져 있다.
- 오른쪽 40% 구간은 같은 막대 안에서 옅은 색으로 표현된다.
- 진행률을 `30`으로 바꾼 뒤 돌아오면 진한 영역이 왼쪽 30%로 줄어 있다.

---

## J15. 간트에서 Overdue 시각 표시

**Given** J9와 동일 조건 ("킥오프 미팅"이 overdue 상태).

**When** 간트 뷰에서 해당 작업 행의 막대를 본다.

**Then**
- 막대에 **경고색 테두리**가 있거나 **빗금 오버레이**가 얹혀 있어, 일반 막대와 시각적으로 구분된다.
- 상태를 "완료"로 바꾸면 경고 표시가 사라진다.

---

## J16. 간트는 읽기 전용

**Given** 간트 뷰를 보고 있다.

**When**
- 막대를 마우스로 드래그해 이동시키려 시도.
- 막대 끝을 잡아 리사이즈하려 시도.

**Then**
- 드래그·리사이즈 둘 다 **작동하지 않는다**.
- (힌트 토스트 노출은 선택) "일정 편집은 목록 뷰의 편집 모달에서 가능합니다" 안내가 나오면 더 좋지만 필수는 아님.

---

## J17. 목표 기한이 시작일보다 앞이면 저장 실패

**Given** 신규 작업 생성 모달.

**When**
1. 시작일 `2026-05-10` 입력
2. 목표 기한 `2026-05-05` 입력
3. **저장** 클릭

**Then**
- 저장되지 않는다.
- 목표 기한 입력 필드 아래에 오류 메시지가 보인다: "목표 기한은 시작일 이후여야 합니다".
- 한쪽만 입력된 경우(시작일만, 또는 목표 기한만)는 오류가 아니다 — 저장 가능.

---

## J18. 드래그 앤 드랍으로 작업 순서 변경

**Given** 목록 뷰에 최상위 작업 "킥오프 미팅"과 "리서치"가 이 순서로 있다.

**When**
1. "리서치" 행 왼쪽의 드래그 핸들에 마우스를 올린다.
2. 핸들을 잡아 "킥오프 미팅" 위치 위로 드랍한다.

**Then**
- 순서가 바뀌어 "리서치"가 먼저, "킥오프 미팅"이 그 아래에 보인다.
- 페이지를 새로고침해도 바뀐 순서가 유지된다.
- 간트 뷰로 전환해도 동일한 순서로 표시된다.
- 다른 부모를 가진 행 위로 드랍하면 순서가 바뀌지 않는다.

## Playwright MCP 수동 회귀 체크리스트

배포 직후나 UI에 큰 변경이 있었던 PR을 리뷰할 때, 아래 순서로 빠르게 훑는다. 실제 Playwright 자동화 테스트가 있으면 이 체크는 보조로만.

```
[ ] J1   빈 화면 렌더
[ ] J2   최상위 작업 추가 → 목록 반영
[ ] J3   하위 작업 추가 → 계층 표시
[ ] J4   접기/펼치기
[ ] J5   진행률 100 → 상태 자동 완료
[ ] J6   인라인 상태 전환
[ ] J7   제목 수정 즉시 반영
[ ] J8   부모 삭제 시 하위 개수 안내 후 함께 삭제
[ ] J9   목록 뷰 Overdue 경고
[ ] J10  CSV 내보내기 다운로드 + 헤더 확인
[ ] J11  CSV 가져오기 정상 (계층 연결 포함)
[ ] J12  CSV 가져오기 부분 오류 (제외/경고 분리 확인)
[ ] J13  간트 뷰 렌더 + 오늘 강조선
[ ] J14  간트 진행률 채움 비율
[ ] J15  간트 Overdue 시각 표시
[ ] J16  간트 드래그 무반응
[ ] J17  시작일 > 목표 기한 검증
[ ] J18  드래그 앤 드랍 순서 변경 + 새로고침 후 유지 + 간트 동기화
```

## 이 문서 확장 규칙

- 새 기능을 `SPEC.md`에 추가했다면, 반드시 이 문서에도 대응 시나리오를 **추가한 뒤** 구현한다 (테스트 먼저).
- 기존 시나리오의 문구를 수정해야 한다면 이유를 커밋 메시지에 남긴다 (회귀 범위 파악용).
- 시나리오 ID(J1, J2 …)는 **재사용·재부여하지 않는다.** 삭제되더라도 번호는 빈 자리로 둔다 (이전 테스트/PR 참조 보존).
