# 제품 스펙 (Product Specification)

> 이 문서는 WBS 앱의 **사용자 관점 기능 스펙의 단일 원천**입니다. 기술 용어(스키마·마이그레이션·컴포넌트 이름 등)는 다루지 않으며, "사용자가 화면에서 무엇을 할 수 있고 무엇을 기대하는가"만 씁니다.
>
> 새 기능을 구현하거나 동작을 바꾸기 전에 반드시 이 문서를 먼저 확인하고, 충돌이 있으면 **코드보다 이 문서를 먼저** 갱신하세요.

---

## 0. 제품 한 줄 요약

엑셀로 관리하던 WBS(Work Breakdown Structure)를 브라우저 하나로 옮기되, 엑셀의 자유도(복붙·CSV 주고받기)와 전용 툴의 시각화(간트)를 둘 다 얻는다.

## 0-1. 대상 사용자

**프로젝트 매니저(PM) 1인.** 소규모 팀 프로젝트의 작업을 쪼개고 진행 상황을 시각화해 관리하고 싶은 사람. 다중 로그인·권한 구분·협업 기능은 범위 밖입니다.

---

## 1. 기능 A — 작업 목록 (Task List)

사용자가 앱을 열면 바로 만나는 메인 화면.

### A-1. 빈 상태
앱을 처음 열면 "아직 작업이 없습니다. 첫 작업을 추가해 시작하세요" 안내와 **+ 작업 추가** 버튼이 보입니다.

### A-2. 작업 행
각 작업은 한 줄로 표시되며, 다음 정보가 한눈에 들어옵니다:

- **제목**
- **담당자** (없으면 회색 "—")
- **상태 배지** — 할 일 · 진행 중 · 완료
- **진행률** — 0~100%, 숫자와 얇은 진행 바
- **시작일 · 목표 기한** — 둘 다 선택적. 하나만 있거나 둘 다 없을 수 있음

### A-3. 순서 변경
사용자가 작업 행의 드래그 핸들을 잡아 같은 부모 내 다른 위치로 드랍하면 순서가 바뀝니다. 변경된 순서는 저장되며 목록 뷰와 간트 뷰에 동일하게 반영됩니다. 서로 다른 부모를 가진 작업 사이의 이동(부모 변경)은 지원하지 않습니다.

### A-4. 화면 목업

```text
┌────────────────────────────────────────────────────────────────────────────┐
│ WBS   [ 목록 | 간트 ]           [ + 작업 추가 ] [ CSV 내보내기 ] [ CSV 불러오기 ] │
├─────┬──────────────────────────┬────────┬─────────┬───────┬─────────────────┤
│     │ 제목                     │ 담당자 │ 상태    │ 진행률│ 기간            │
├─────┼──────────────────────────┼────────┼─────────┼───────┼─────────────────┤
│  ▼  │ 킥오프 미팅              │ 김PM   │ [진행중]│  60%  │ 5/1 ~ 5/3 [지남]│   ← "지남" 배지와
│     │    아젠다 초안 작성      │ 박기획 │ [완료]  │ 100%  │ 5/1 ~ 5/2       │     날짜 텍스트는 빨강
│  ▶  │ 리서치                   │   —    │ [할 일] │   0%  │ 5/8 ~ 5/14      │
│     │ 디자인 리뷰              │ 이PM   │ [할 일] │   0%  │       —         │
└─────┴──────────────────────────┴────────┴─────────┴───────┴─────────────────┘
   ↑                                  ↑                            ↑
 펼침/접힘 아이콘                   배지 클릭 시                 시작·기한 없으면
 (자식이 없으면 없음)                인라인 순환 전환              "—" 표시
```

## 2. 기능 B — 작업 생성

### B-1. 진입점
- 목록 상단의 **+ 작업 추가** 버튼 → 최상위 작업으로 생성
- 특정 작업 행의 **⋯ 메뉴 → 하위 작업 추가** → 해당 작업의 자식으로 생성

### B-2. 입력 필드
모달 또는 인라인 폼에서 다음을 입력합니다.

| 필드 | 필수 여부 | 비고 |
|---|---|---|
| 제목 | 필수 | |
| 설명 | 선택 | 여러 줄 |
| 담당자 | 선택 | 자유 텍스트 (이름 또는 핸들) |
| 상태 | 선택 | 기본값 "할 일" |
| 진행률 | 선택 | 0~100, 기본값 0 |
| 시작일 | 선택 | 날짜 선택기 |
| 목표 기한 | 선택 | 날짜 선택기. **시작일 이후**여야 함 |
| 상위 작업 | 자동 | 하위 작업 추가로 진입하면 자동 지정 |

### B-3. 저장 후
모달이 닫히고, 목록에 새 행이 즉시 반영됩니다.

## 3. 기능 C — 작업 수정

### C-1. 진입점
목록에서 **작업 행 클릭** → 편집 모달.

### C-2. 수정 규칙
- 모든 필드를 수정할 수 있습니다.
- **상태 배지는 인라인 클릭**만으로도 할 일 ↔ 진행 중 ↔ 완료 순환 전환이 가능합니다(모달을 열지 않아도 됨).
- **진행률을 100으로 올리면 상태는 자동으로 "완료"**가 됩니다(편의).
- **역방향 자동 동기화는 없음** — 상태를 바꿨다고 해서 진행률이 따라 바뀌지 않습니다. "완료" → "진행 중"으로 내려도 진행률은 그대로, "할 일" → "완료"로 올려도 진행률이 자동으로 100이 되지 않습니다. 진행률은 항상 사용자가 직접 조정.

---

## 4. 기능 D — 작업 삭제

### D-1. 진입점
작업 행의 **⋯ 메뉴 → 삭제**.

### D-2. 확인 다이얼로그
"이 작업을 삭제합니다. 계속할까요?"

하위 작업이 있으면 문구가 바뀝니다:

> "이 작업과 하위 작업 N개가 모두 삭제됩니다. 계속할까요?"

확인 시 해당 작업과 모든 하위 작업이 함께 사라집니다.

---

## 5. 기능 E — 계층(Hierarchy)

### E-1. 부모-자식 표현
작업은 다른 작업의 하위로 들어갈 수 있습니다. 목록에서는 **들여쓰기**로 표현하며, 자식을 가진 행의 왼쪽에 펼침/접힘 아이콘이 붙습니다.

### E-2. 펼침/접힘
- 펼친 상태: **▼**
- 접힌 상태: **▶**
- 자식이 없으면 아이콘 없음
- 토글은 아이콘 클릭.

### E-3. 깊이
깊이에는 강제 상한이 없지만, 실사용상 3~4단계 이상은 권장하지 않습니다 (UX 가이드라인일 뿐 시스템 제한은 아님).

## 6. 기능 F — CSV 가져오기 / 내보내기

엑셀에서 편집해 가져오는 유스케이스를 위한 핵심 기능.

### F-1. 내보내기
툴바 **CSV 내보내기** → 현재 목록 전체(계층 포함)를 CSV 파일로 다운로드.

- 파일명 패턴: `wbs-YYYY-MM-DD.csv`
- 열 순서(헤더): `제목, 설명, 담당자, 상태, 진행률, 시작일, 목표 기한, 상위 작업 제목`
- 같은 제목의 부모가 여러 개라면 동명 처리 규칙: MVP에서는 그냥 먼저 매칭되는 것 사용(가져오기 시).

### F-2. 가져오기
툴바 **CSV 불러오기** → 파일 선택 → **미리보기** → **적용**.

- 미리보기는 "N개 작업을 추가합니다. 제외 M건"을 보여주고, 제외 사유를 행 단위로 나열합니다.
- 제외 규칙:
  - 제목이 비면 건너뜀
  - 날짜 형식 불량이면 해당 날짜만 비움
  - 상태 값이 허용 외이면 "할 일"로 대체
  - 상위 작업 제목이 목록·CSV 내에서 매칭되지 않으면 **최상위**로 추가
- **추가만 수행합니다.** 기존 작업을 수정·삭제하지 않습니다(MVP 단순화).

## 7. 기능 G — 간트 일정 시각화

엑셀 WBS 우측의 "일정 표"를 웹으로 옮긴 뷰. 좌측엔 작업 트리, 우측엔 주 단위 날짜 그리드 위에 각 작업의 기간을 가로 막대로 그립니다.

### G-1. 뷰 전환
상단 토글로 **목록 뷰 ↔ 간트 뷰** 전환.

### G-2. 화면 구성
- **좌측**: 작업 트리 (목록 뷰와 동일한 계층·들여쓰기)
- **우측**: 가로 날짜 그리드 (**주 단위** 눈금, 오늘 날짜에 세로 강조선)
- **각 작업 행의 막대**: 시작일 ~ 목표 기한 구간을 하나의 가로 막대로 표시
- **진행률 표현**: 막대 내부 왼쪽부터 진행률(%)만큼 **진한 색**, 나머지는 옅은 색
  - 예: 진행률 60%면 막대 왼쪽 60%가 진한 색, 오른쪽 40%가 옅은 색
- **시작일 또는 목표 기한이 비어 있는 작업**: 막대 없이 "— 일정 없음 —" 표기

### G-3. 편집 불가 (MVP)
간트 뷰는 **읽기 전용**입니다. 막대를 드래그해도 기간이 바뀌지 않습니다. 편집은 목록 뷰의 편집 모달에서만.

### G-4. 화면 목업

오늘이 `2026-05-14` 라고 가정했을 때:

```text
                                                   오늘(5/14)
                                                       │
                                                       ▼
┌──────────────────────────┬────────── 2026년 5월 ──────────────┬──── 6월 ─────┐
│ 작업               진행률│ 5/1  5/8  5/15 5/22 5/29           │ 6/5   6/12   │
├──────────────────────────┼────────────────────────────────────┼──────────────┤
│ ▼ 킥오프 미팅        60% │ ████████▒▒▒▒                       │              │
│      아젠다 초안    100% │ ██████                             │              │
│ ▶ 리서치              0% │           ░░░░░░░░░░░░             │              │
│   개발 착수          20% │                        ██▒▒▒▒▒▒▒▒▒▒│▒▒▒▒          │
│   디자인 리뷰         —  │               — 일정 없음 —          │              │
└──────────────────────────┴────────────────────────────────────┴──────────────┘

 범례:
   ████   진행 완료 영역 (진한 색, 막대 왼쪽에서 진행률만큼 채움)
   ▒▒▒▒   남은 예정 영역 (같은 막대의 나머지, 옅은 색)
   ░░░░   진행률 0% 작업의 막대 (전체 옅은 색)
   │      오늘 날짜 세로 강조선 (그리드를 가로지름)
   —      시작일/목표 기한 중 하나라도 비면 막대 없이 "일정 없음"
```

읽는 방법(위 목업의 각 행이 어떤 상태를 나타내는지):

- **킥오프 미팅** (진행률 60%) → 막대 왼쪽 60% 진한 색, 오른쪽 40% 옅은 색.
- **아젠다 초안** (100%) → 짧은 막대 전부 진한 색.
- **리서치** (0%) → 전체가 옅은 색(아직 시작 전).
- **개발 착수** (20%, 기간이 월 경계를 건넘) → 왼쪽 20%가 진한 색, 막대가 5월/6월 사이 세로선을 가로질러 걸쳐 있음.
- **디자인 리뷰** (일정 없음) → 막대 없이 트랙 안에 "— 일정 없음 —"만 표시.

> 날짜 숫자는 ASCII 해상도의 한계로 픽셀 단위가 맞지 않을 수 있습니다. 실제 구현에서는 시작일/목표 기한 대비 정확한 위치와 폭으로 막대를 그립니다.

## 8. 기능 H — 목표 기한 지남 (Overdue) 시각 표시

### H-1. 판정 기준
**오늘 날짜가 목표 기한보다 뒤**이고 **상태가 "완료"가 아닌** 작업을 "지남(Overdue)"으로 간주합니다.

### H-2. 표시 방법

**목록 뷰**
- 목표 기한 날짜 텍스트가 **경고색(빨강 계열)** 으로 표시
- 날짜 옆에 **"지남" 배지** 부착

**간트 뷰**
- 해당 작업의 막대에 **경고색 테두리** 또는 **빗금 오버레이**

### H-3. 상태가 "완료"가 되면
즉시 경고 표시와 배지가 사라집니다.

### H-4. 화면 목업

오늘이 `2026-05-14`, "킥오프 미팅"의 목표 기한은 `2026-05-03`, 상태는 `[진행중]` 이라고 가정.

**목록 뷰에서**

```text
┌─────┬──────────────────┬────────┬─────────┬───────┬─────────────────┐
│     │ 제목             │ 담당자 │ 상태    │ 진행률│ 기간             │
├─────┼──────────────────┼────────┼─────────┼───────┼─────────────────┤
│  ▼  │ 킥오프 미팅      │ 김PM   │ [진행중]│  60%  │ 5/1 ~ 5/3 [지남] │  ← 빨강 + 배지
│     │   아젠다 초안    │ 박기획 │ [완료]  │ 100%  │ 5/1 ~ 5/2        │  (완료라 경고 없음)
└─────┴──────────────────┴────────┴─────────┴───────┴─────────────────┘
                                                        │        │
                                                 날짜 텍스트    "지남" 배지
                                                 경고색(빨강)   회색이 아닌 빨강
```

**간트 뷰에서 — 같은 작업이 overdue로 강조됨**

overdue 표시 상자(╔═╗/╚═╝)는 **막대 전체(채움 + 남은 부분)를 함께 감쌉니다.** 일반 막대와의 차이는 "주변을 둘러싼 빨강 테두리(+ 빗금 오버레이)"뿐이며, 내부 진행률 표현 규칙은 동일합니다.

```text
                                            오늘(5/14)
                                                │
                                                ▼
┌─────────────────────┬──────── 2026년 5월 ──────────────┬──── 6월 ────┐
│ 작업          진행률│ 5/1  5/8  5/15 5/22 5/29         │ 6/5   6/12  │
├─────────────────────┼──────────────────────────────────┼─────────────┤
│ ▼ 킥오프 미팅   60% │╔═══════╗                         │             │  ← 막대 전체에
│                     │║█████▒▒║                         │             │    빨강 테두리
│                     │╚═══════╝                         │             │    (+ 빗금 가능)
│    아젠다 초안 100% │ ██████                           │             │
│ ▶ 리서치         0% │           ░░░░░░░░░░             │             │
└─────────────────────┴──────────────────────────────────┴─────────────┘

 │ █████ │  = 채움 60% (진한 색)
 │    ▒▒ │  = 남은 40% (옅은 색, 같은 막대 내부)
 │╔═══╗  │  = overdue 강조: 위 채움/남은 영역 둘 다 이 상자 안에 들어감
```

위 두 뷰에서 같은 작업을 상태 `[완료]`로 바꾸면, 빨강 날짜·`[지남]` 배지·빨강 테두리가 **모두 즉시 사라지고** 일반 막대/일반 글자색으로 돌아갑니다.

## 9. 기능 I — 산출물(Deliverables) 관리

작업마다 그 작업이 만들어내는 결과물(문서·디자인·코드 등)을 기록하고, 완료 여부를 추적합니다.

### I-1. 산출물이란

작업 하나가 "완료됐다"고 말할 수 있으려면 이 파일·문서·코드가 실제로 나와야 한다는 목록입니다.

### I-2. 산출물의 속성

| 속성 | 설명 |
|---|---|
| 이름 | 산출물을 가리키는 짧은 이름 (예: 화면 설계서, API 명세) |
| 유형 | DOC · DSN · CODE · SPEC · DATA · ETC 중 하나 |
| 상태 | 대기 · 진행 중 · 완료 중 하나, 기본 "대기" |
| 링크 | 선택적 URL (Google Docs, Figma, GitHub 등) |

### I-3. 작업 생성 시 산출물 추가

작업 생성 모달 하단에 "산출물" 섹션이 있으며, 유형 + 이름을 입력 후 **+ 추가** 버튼(또는 Enter)으로 목록에 올릴 수 있습니다. 작업이 저장될 때 함께 생성됩니다. 저장 전에 제거도 가능합니다.

### I-4. 편집 모달에서 산출물 관리

작업 편집 모달에서 산출물 섹션:
- **추가**: 유형 + 이름 입력 → **+ 추가** (즉시 저장됨)
- **상태 전환**: 상태 배지를 클릭하면 대기 → 진행 중 → 완료 → 대기 순환 (즉시 저장됨)
- **삭제**: 각 항목 오른쪽의 × 버튼 (즉시 저장됨)

산출물 변경은 "저장" 버튼과 무관하게 클릭과 동시에 서버에 반영됩니다.

### I-5. 목록 뷰에서의 요약 표시

작업 행에 "산출물" 열이 추가됩니다. 각 산출물의 상태를 작은 컬러 점으로 나열하고, "완료 수/전체 수"를 함께 표시합니다. 산출물이 없으면 "—"으로 표시합니다.

- 회색 점: 대기
- 파란 점: 진행 중
- 초록 점: 완료

## 10. 범위 밖 (Out of Scope)

아래는 MVP에서 구현하지 않습니다. Claude는 사용자의 명시적인 요청 없이는 이 항목들을 추가하지 않습니다.

- 다중 사용자 로그인, 권한, 공유, 초대
- 작업 간 **의존성(선후행)**, 크리티컬 패스
- 간트에서의 **드래그 편집**, 리사이즈, 리스케줄링
- 실시간 협업, 변경 알림(Email/Push/Slack)
- 반복 작업, 작업 템플릿
- 첨부 파일, 코멘트, 활동 로그
- 사용자 정의 필드, 커스텀 상태
- 자동 정렬·검색·필터·저장된 뷰 (수동 드래그 순서 변경은 범위 안)
- 모바일 앱 (반응형 웹은 범위 안 — 단, 데스크탑 우선)

새 기능을 이 목록에서 빼서 "범위 안"으로 옮기려면 본 문서를 먼저 수정한 뒤 구현을 시작하세요.
